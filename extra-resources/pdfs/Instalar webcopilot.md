A lo sumo son 15 min de arreglar boludeces por una tool que (al menos por lo que la llevo probando) vale la pena.

Si están en Parrot o Kali (si están en otras distros tienen que mirar el install.sh y corregir lo que corresponda) el paso a paso es el siguiente:

1) En el directorio que quieran (yo lo hice en /opt) se convierten en root (es necesario convertirse, no usar sudo nomás).
2) `git clone https://github.com/h4r5h1t/webcopilot && cd webcopilot/ && chmod +x webcopilot install.sh && mv webcopilot /usr/bin/ && ./install.sh` corren este comando de una.
3) Van a empezar a ver un montón de logs pero lo único importante (si todo va bien, si les aparecen errores ya será cosa de ver cada error particular) es ver los casos en los que se lea "Please install it manually". Les va a aparecer que algunas tools no se pudieron instalar. Tomen nota de cuáles y cuando el script termine de correr las instalan manualmente.
4) En mi caso tuve problemas con aquatone y dnsx. Para dnsx usando `apt install dnsx` se soluciona (estando todavía como root). 
5) Para aquatone: [michenriksen/aquatone: A Tool for Domain Flyovers](https://github.com/michenriksen/aquatone) seguir los pasos de Installation: descargar la última release (en este momento 
aquatone_linux_amd64_1.7.0.zip, chequeen que sea la amd64), yo la metí en Downloads. Se mueven al directorio donde lo bajaron y hacen `unzip aquatone_linux_amd64_1.7.0.zip`. Les van a quedar 4 archivos en total, pueden borrar todo menos el que se llama aquatone. O sea, dejen solo el ejecutable. Después hacen `mv aquatone /usr/bin/aquatone` (como root o sudo) o lo meten en la ruta que quieran pero que esté en su PATH (`echo $PATH` para verlo). /usr/bin sí o sí está, así que lo mejor es meterlo ahí si no quieren pensarla tanto. Con eso ya está instalado aquatone.

Una vez hechos estos pasos ya están instaladas las tools. Ahora sí quitando eso quedan arreglar dos cosas nomás:

1) El escaneo de subdominios corre `amass`. La versión 4.2.0 de amass cambió el output que devuelve y es un toque más bardo usarlo. Les recomiendo que hagan esto `amass -version` y si ven 4.1.0 ignoren lo que respecta a amass, debería andarles bien. Si ven 4.2.0 que sería lo más normal, pueden hacer lo siguiente: [opción 1] se mueven a /usr/bin/ y como root o con sudo editan el webcopilot: `sudo nano webcopilot` (o lo abren con un editor con interfaz gráfica desde el explorador de archivos, como quieran. Si usan nano se guarda con Ctrl+S y se cierra con Ctrl+X). Empiezan a bajar hasta la parte de Passive Subdomain Enumeration y comentan la línea de `[ ! -f temp/passive_amass.txt ] && amass enum -passive -norecursive -d $domain -o temp/passive_amass.txt &> /dev/null` o sea le ponen un # adelante. Lo mismo con la línea `[ ! -f temp/active_amass.txt ] && timeout 22m amass enum -active -brute -w ~/wordlists/subdomains.txt -d $domain -o temp/active_amass.txt &> /dev/null` pero de Active Subdomain Enumeration. Es decir, salteamos totalmente el escaneo de amass. No se va a ejecutar. [opción 2] arreglan el script para que funcione correctamente con amass. La única solución que encontré está acá [Exclude ASN and Netblocks from output · Issue #1023 · owasp-amass/amass](https://github.com/owasp-amass/amass/issues/1023) específicamente el comentario de capastacka. Probé correr amass con esa solución y digamos que "funcionó" (no dentro del script, aparte). No recomiendo que hagan esto igual a menos que encuentren forma de que el amass realmente funcione correctamente. [opción 3] se me ocurre que otra opción es lograr downgradear la versión de amass a la 4.1.0 o menos, pero no lo probé. Lo menciono igual.
2) Lo último es poner 2 API Keys que son necesarias para que la tool funcione correctamente. La solución es esta: [another SUBLIST3R 2.0 STUCK ......... did someone manage to make it work? · Issue #26 · h4r5h1t/webcopilot](https://github.com/h4r5h1t/webcopilot/issues/26) pero les dejo el paso a paso acá igual. [1] tienen que crear cuentas en [VirusTotal - Home](https://www.virustotal.com/gui/home/upload) y [DNSDumpster - Find & lookup dns records for recon & research](https://dnsdumpster.com/). Una vez que tienen sus cuentas busquen en Settings o en donde sea su API Key (ya con loggear las aparece igual creo). Van a ser 2 valores alfanuméricos larguitos. Se los anotan o los tienen a mano. [2] **como root** tienen que copiarlos dentro de su archivo rc de su shell. Para eso primero vean qué shell están usando `echo $SHELL` o hagan `cat /etc/passwd` y vean la primera línea qué dice (como root, tienen que ver la shell de root). Lo más probable es que vean bash o zsh. Tienen que hacer lo mismo para cualquier caso (si no es zsh o bash capaz tengan que hacer algo distinto). [3] Si dice bash, como root, editan el .bashrc. Si dice zsh, como root, editan el .zshrc. Para eso hacen `nano ~/.zshrc` o `nano ~/.bashrc` (como root sí o sí) y suman las siguientes líneas en 2 líneas distintas: ```export VTAPIKEY="Your_VirusTotal_API_Key_Here"  
export DNSDUMPSTER_API_KEY="Your_DNSdumpster _API_Key_Here"``` reemplazando dentro de las comillas sus API keys correspondientes de cada página. Una vez hecho eso Ctrl+S para guardar Ctrl+X para cerrar el editor. [4] Corren `source ~/.bashrc` o `source ~/.zshrc` lo que les corresponda y listo.

Ya con eso debería funcionarles la tool. Para correrla usan (siendo root, no con sudo adelante) `webcopilot -d target.com` con el dominio que quieran y listo! Por defecto solamente corre enumeración de subdominios, pero con el -a creo que corre escaneo de vulns también. Cualquier cosa vean con `webcopilot -h` todas las opciones.
